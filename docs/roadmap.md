# Roadmap — God Simulator

## Философия

Сначала модель — потом картинка. Каждый этап должен быть интересен сам по себе как симуляция, прежде чем двигаться дальше. Визуал улучшается только тогда, когда логика под ним стабильна.

---

## Обзор

```
Python (симуляция)                         Godot (игра)
─────────────────────────────────────────────────────────────────
MVP 1 → MVP 2 → MVP 3 → MVP 4 │ Migrate → MVP 5 → MVP 6 → MVP 7
симул.  ресурсы  строит.  бог  │  годот    соц.    эпохи   релиз
```

Граница перехода: когда симуляция достаточно богата чтобы в неё было интересно играть, но ещё нет финального визуала — переезжаем на Godot и там уже доводим до игры.

---

## MVP 1 — Живая симуляция
**Стек:** Python + Pygame (top-down, цветные квадраты, белые точки)

**Цель:** агенты живут, умирают, размножаются без участия игрока. Симуляция выглядит живой.

**Что входит:**
- Карта 1000×1000, тайловая, процедурная генерация (seed)
- 10 агентов (5М+5Ж), полный жизненный цикл
- 7 потребностей: hunger, thirst, energy, health, mood, social, safety
- 5 черт: fearfulness, sociality, curiosity, resilience, faith
- Utility-based AI + стохастика через curiosity
- Память агентов, радиус обзора 10 тайлов
- Время суток (день/ночь)
- A* патфайндинг с кешем
- Рождение, смерть, усыновление
- Система событий с реакцией через черты
- UI: viewport, клик на агента, лог, контроль скорости
- **Инструментарий:** headless runner, batch runner, analyze.py с автофлагами, timeline scrubber

**Критерий готовности:**
- 70%+ прогонов выживают до тика 50 000
- Популяция стабилизируется в коридоре 8–30 агентов
- Ни один флаг не горит красным на 10+ seeds подряд

## MVP 2 — Ресурсное давление
**Стек:** Python + Pygame

**Цель:** поселение испытывает давление среды, появляется причина расширяться. Популяция регулируется ресурсами, а не только параметрами размножения.

**Что добавляется:**
- Конечные ресурсы: лес истощается, восполняется со временем
- Агенты замечают истощение и ищут новые источники
- Простое "освоение территории" — агенты помечают зоны как истощённые
- Внешнее давление: случайные болезни (health начинает реально работать)
- Балансировка: численность популяции стабилизируется через ресурсное давление

**Новые автофлаги:**
- `RESOURCE_DEADLOCK` — все известные агентам ресурсы истощены одновременно
- `MIGRATION_FAILURE` — агенты не находят новые ресурсы при истощении старых

**Критерий готовности:**
- Популяция колеблется органично вокруг стабильного числа
- Видна миграция к новым ресурсам в timeline
- Болезни заметны в логе, но не убивают всех

---

## MVP 3 — Строительство и труд
**Стек:** Python + Pygame

**Цель:** агенты начинают менять среду. Появляется понятие "поселение" — что-то что можно увидеть на карте и узнать через 100 тиков.

**Что добавляется:**
- Базовые постройки: склад, жилище, колодец
- Труд как действие: собирать, носить, строить
- Агенты строят по потребности (нет жилья → строят)
- Хранение ресурсов (склад снижает decay hunger/thirst)
- Зачаточное разделение труда через черты (высокий resilience → чаще строит)

**Открытые вопросы которые нужно решить:**
- Как агенты "решают" что строить — коллективное или индивидуальное решение?
- Нужны ли явные роли или emergent через черты?

**Критерий готовности:**
- За несколько часов симуляции на карте появляются постройки
- Поселение с постройками выживает заметно дольше чем без

## MVP 4 — Бог
**Стек:** Python + Pygame

**Цель:** игрок получает инструмент влияния. Первая настоящая механика игры. Проверяем что непрямое управление работает и это интересно.

**Что добавляется:**
- Действия бога: наказать (молния/болезнь), поощрить (еда/исцеление/радость)
- Ресурс бога (мана/вера) — нельзя спамить
- Система интерпретации:
  - Действие бога = событие с большим base_impact
  - Агент пропускает через черты → меняет поведение
  - fearfulness × impact = паника или ультранабожность
  - Ошибочная интерпретация: вероятностная привязка к контексту события
- Вера как коллективный стат поселения
- Ресурс бога пополняется от веры — петля обратной связи

**Открытые вопросы:**
- Как формализовать ошибочную интерпретацию
- Как вера меняет поведение агентов (ритуалы? жертвоприношения?)
- Как игрок понимает что агенты "поняли правильно"

**Критерий готовности:**
- За 15 минут наблюдения можно увидеть как действия бога меняют поведение
- Иногда не так как ожидал — это фича, не баг
- Есть хотя бы одна петля обратной связи: вера → мана → действие → вера

## Переезд на Godot
**После:** MVP 4 завершён и принят

**Почему именно здесь:**
- К этому моменту вся симуляционная логика стабильна и задокументирована
- Параметры засетаплены через сотни прогонов batch runner
- Есть понимание что в игре происходит и как это должно выглядеть
- Дальше нужны изометрия, анимации, нормальный UI — Python/Pygame для этого не подходит

**Что происходит при переезде:**
- Симуляционное ядро (ECS, системы, компоненты) переписывается на GDScript или C#
- Python спека служит точным референсом — никаких угадываний
- Batch runner и analyze.py остаются на Python для валидации: прогоняем те же seeds, сравниваем stats.jsonl — поведение должно совпасть
- Top-down вид на первом этапе переезда — убеждаемся что симуляция работает идентично
- Изометрия подключается после валидации

**Риски:**
- GDScript медленнее Python на чистых вычислениях — при 500+ агентах может потребоваться C# или GDNative
- Headless batch runner в Godot менее удобен — возможно часть аналитики останется на Python навсегда

## MVP 5 — Социальные структуры (Godot)
**Стек:** Godot, top-down → изометрия

**Цель:** из агентов органически возникают группы и роли. Появляется ощущение общества.

**Что добавляется:**
- Отношения между агентами (friendship, rivalry, trust)
- Семьи — родители помнят детей, дети наследуют связи
- Лидерство — emergent, через stats и доверие других
- Простые нормы поведения — правила которые поселение само себе устанавливает через опыт
- Конфликты между агентами
- Переключение на изометрический вид если симуляция валидирована

---

## MVP 6 — Эпохи и прогресс (Godot)
**Цель:** поселение движется по времени, появляется ощущение истории.

**Что добавляется:**
- Технологический прогресс как накопление знаний
- Смена эпох: племя → деревня → город
- Каждая эпоха открывает новые постройки и поведение
- Письменность — начинается история, появляются записи внутри игры
- Внешние события: засуха, нашествие, открытие нового ресурса

---

## MVP 7 — Полировка и сценарии (Godot)
**Цель:** из симуляции делаем игру.

**Что добавляется:**
- Сценарии с целями (выжить N лет, достичь эпохи X, сохранить популяцию при катастрофе)
- Офлайн симуляция (тики идут пока не играешь)
- Нормальный UI, звук, музыка
- Туториал

## Принципы на весь roadmap

**Мало правил — сложное поведение.** Каждый этап добавляет небольшой набор механик которые взаимодействуют с существующими. Не добавлять правила ради правил — только если реально меняют поведение.

**Сначала симуляция — потом баланс.** Механика должна работать прежде чем её балансировать. Балансировка изолированно, через batch runner.

**Тестируемость не обсуждается.** Каждая новая механика должна покрываться автофлагами до того как считается готовой.

**Python → Godot — не переписывание, а перенос.** Python спека это живой документ который описывает поведение. Godot реализация должна воспроизводить те же stats.jsonl на тех же seeds.
