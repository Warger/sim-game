# Тестирование и аналитика — MVP 1

## Проблема

Симуляция с 10 агентами на 1000×1000 карте — это тысячи решений за минуту. Вручную проверять "живая ли симуляция" по экрану медленно и ненадёжно. Нужен инструмент который сам скажет что работает, что сломано и где нужно покрутить параметры.

---

## Слой 1 — Структурированный лог

Каждый прогон симуляции пишет JSON лог. Не plaintext — именно JSON, чтобы его можно было парсить программно.

### Структура лог-файла

```
logs/
  run_{seed}_{timestamp}/
    meta.json          — параметры запуска
    events.jsonl       — одно событие на строку (JSON Lines)
    agents.jsonl       — снимок состояния агентов каждые N тиков
    deaths.jsonl       — детальная запись каждой смерти
    births.jsonl       — детальная запись каждого рождения
    stats.jsonl        — агрегированные метрики каждые N тиков
```

### meta.json
```json
{
  "seed_map": 42,
  "seed_agents": 7,
  "start_agents": 10,
  "ticks_run": 50000,
  "real_time_seconds": 12.4,
  "config_snapshot": { ...все значения config.py... }
}
```

### events.jsonl (одна строка = одно событие)
```json
{"tick": 1240, "type": "death", "agent_id": 3, "name": "Aria", "age_ticks": 48200, "cause": "thirst", "children": 2}
{"tick": 1241, "type": "birth", "agent_id": 11, "name": "Beron", "mother_id": 2, "father_id": 5}
{"tick": 1242, "type": "faint", "agent_id": 7, "name": "Cela", "cause": "hunger"}
```

### agents.jsonl (снимок каждые 100 тиков)
```json
{"tick": 100, "agent_id": 1, "name": "Aria", "age": 158000, "sex": "F", "stage": "adult",
 "needs": {"hunger": 0.72, "thirst": 0.54, "energy": 0.88, "health": 1.0, "mood": 0.61, "social": 0.44, "safety": 0.91},
 "traits": {"fearfulness": 0.38, "sociality": 0.71, "curiosity": 0.55, "resilience": 0.62, "faith": 0.48},
 "action": "eating", "pos": [512, 498]}
```

### stats.jsonl (агрегат каждые 100 тиков)
```json
{"tick": 100, "population": 9, "alive_adults": 7, "alive_children": 2,
 "avg_hunger": 0.68, "avg_thirst": 0.71, "avg_mood": 0.55,
 "deaths_last_100": 1, "births_last_100": 0,
 "actions_distribution": {"eating": 3, "drinking": 1, "sleeping": 2, "wandering": 2, "socializing": 1}}
```

## Слой 2 — Headless runner

Симуляция без UI. Работает в 10–50x быстрее чем с рендером.

```bash
# Один прогон
python main.py --headless --seed 42 --ticks 50000

# Прогон до вымирания или до максимума тиков
python main.py --headless --seed 42 --ticks 200000 --stop-on-extinction
```

### Batch runner — N симуляций автоматически

```bash
python tools/batch_run.py --seeds 10 --ticks 50000 --parallel 4
```

- Запускает симуляции на seeds 0–9 (или случайных)
- `--parallel 4` — 4 процесса параллельно
- Каждый прогон пишет свой лог в `logs/`
- По завершении выводит сводку: сколько выжили, средняя популяция, вымершие

## Слой 3 — Аналитический скрипт

```bash
python tools/analyze.py --runs logs/
```

Читает все логи из директории и выводит отчёт.

### Что анализирует

**Выживаемость**
- Сколько % прогонов дожили до тика N
- Медианный тик вымирания (если вымерли)
- Причины вымирания: в основном жажда? голод? не размножились?

**Демография**
- График популяции по тикам (усреднённый по всем прогонам)
- Коридор min/max популяции
- Средний возраст смерти
- Соотношение рождений и смертей по времени

**Потребности**
- Средние значения каждой потребности по времени
- Какая потребность чаще всего критическая
- Есть ли агенты которые постоянно голодают (патфайндинг не работает?)

**Поведение**
- Распределение действий: сколько % времени агенты едят / спят / бродят
- Есть ли агенты которые вообще не социализируются
- Ночное поведение vs дневное

**Аномалии — автоматические флаги**
- 🔴 `EXTINCTION_EARLY` — вымерли до тика 10 000 (проблема с базовыми потребностями)
- 🔴 `NO_BIRTHS` — ни одного рождения за 20 000 тиков (условия размножения слишком жёсткие)
- 🟡 `LOW_POPULATION` — популяция ниже 5 дольше 5 000 тиков (рядом с вымиранием)
- 🟡 `THIRST_CRISIS` — среднее thirst < 0.3 дольше 1 000 тиков (агенты не находят воду)
- 🟡 `STUCK_AGENTS` — агенты с одинаковыми координатами 500 тиков подряд (застряли)
- 🟢 `STABLE` — популяция 8–20 агентов дольше 30 000 тиков (симуляция здорова)

## Слой 4 — Debug inspector (для конкретного агента)

Иногда аномалия видна в aggregate, но непонятно почему. Тогда нужно посмотреть на конкретного агента.

```bash
python tools/inspect_agent.py --run logs/run_42_xxx --agent "Aria"
```

Выводит:
- Полный таймлайн действий агента
- Все точки где он менял решение
- Все броски кубика на curiosity
- График потребностей по времени
- Момент смерти с последними 20 событиями

## Рабочий процесс тюнинга

```
Изменил параметр в config.py
        ↓
python tools/batch_run.py --seeds 20 --ticks 50000
        ↓
python tools/analyze.py --runs logs/
        ↓
Смотришь на флаги и метрики
        ↓
Понял что именно не так → inspect_agent для деталей
        ↓
Правишь → повторяешь
```

Весь цикл занимает 2–5 минут на 20 прогонов по 50 000 тиков.

## Ключевые метрики здоровья симуляции

По этому набору можно быстро сказать "симуляция работает":

| Метрика | Здорово | Проблема |
|---------|---------|---------|
| Выживаемость до тика 50k | > 70% прогонов | < 50% |
| Популяция в тике 50k | 10–30 агентов | < 5 или > 100 |
| Среднее hunger | > 0.5 | < 0.3 |
| Среднее thirst | > 0.5 | < 0.3 |
| Рождений за 50k тиков | > 5 | 0 |
| % времени wandering | 20–40% | > 60% (нечем заняться) или < 5% (застряли) |
| Застрявшие агенты | 0 | > 0 |
